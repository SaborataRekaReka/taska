# architecture.md — архитектура и границы системы

## 1) Назначение документа
Этот документ фиксирует устойчивый архитектурный контекст проекта на “правильной высоте”:
- достаточно конкретно для разработки и эксплуатации;
- без избыточных деталей, которые быстро устаревают.

Цель: чтобы любой инженер или AI-агент быстро понимал **как устроена система, где границы модулей, какие инварианты нельзя ломать**.

---

## 2) Системные контексты

### 2.1 Product Context
Система управления задачами для одного пользователя (MVP), с возможностью расширения до мультипользовательского режима.

Основные сценарии:
- управление списками и задачами;
- подзадачи, приоритеты, дедлайны, статусы;
- история изменений;
- AI-ассистент в safe-mode (только через подтверждение пользователя).

### 2.2 Frontend Context
Ответственность:
- UI/UX, навигация, отображение списков/задач/статистики;
- локальное состояние интерфейса;
- вызовы backend API;
- optimistic updates с корректным откатом при ошибках.

Вне зоны ответственности:
- бизнес-правила владения сущностями;
- проверка прав доступа;
- авторитетные вычисления статистики/истории.

### 2.3 Backend Context
Ответственность:
- API-контракты;
- доменная логика (lists/tasks/history/ai actions);
- авторизация и проверка доступа;
- транзакционная целостность;
- интеграция с БД и внешними сервисами.

Вне зоны ответственности:
- хранение UI-состояния;
- презентационная логика.

### 2.4 Data Context
Ответственность:
- персистентность доменных сущностей;
- референциальная целостность;
- миграции схемы;
- индексы и ограничения.

## 3) Модульная карта (bounded modules)

## 3.1 Auth
Назначение:
- вход/регистрация;
- выдача и проверка токена сессии.

Контракты:
- login/register/logout/me.

Инварианты:
- каждый запрос к приватным данным должен быть привязан к аутентифицированному пользователю.

## 3.2 Users
Назначение:
- профиль пользователя;
- пользовательские настройки (включая язык, предпочтения и пр.).

Инварианты:
- пользователь управляет только своими настройками.

## 3.3 Lists
Назначение:
- CRUD списков;
- порядок списков;
- системные/дефолтные списки.

Инварианты:
- список принадлежит одному пользователю;
- удаление списка не должно приводить к потере задач без явной политики (перенос/архив/каскад по правилам).

## 3.4 Tasks
Назначение:
- CRUD задач;
- статусы, приоритеты, дедлайны;
- фильтры и сортировки;
- подзадачи.

Инварианты:
- задача принадлежит пользователю;
- подзадача не может принадлежать другому пользователю, чем родитель;
- запрещены циклы в иерархии задач.

## 3.5 History
Назначение:
- журнал значимых изменений сущностей;
- основа для undo/audit.

Инварианты:
- история должна быть упорядочена по времени;
- записи истории неизменяемы (append-only).

## 3.6 AI Assistant
Назначение:
- анализ задач;
- подготовка предложений изменений;
- выполнение только подтвержденных действий (safe-mode).

Инварианты:
- AI не применяет destructive операции без подтверждения;
- все AI-изменения должны быть трассируемы;
- должен быть доступен откат последней AI-операции.

## 4) Потоки данных

## 4.1 CRUD-поток (общий)
1. UI инициирует действие.
2. Frontend отправляет запрос в API.
3. Backend валидирует вход и права доступа.
4. Backend применяет доменную логику и сохраняет изменения в БД.
5. Backend возвращает результат.
6. Frontend обновляет состояние и интерфейс.

## 4.2 Поток AI-действий (safe-mode)
1. Пользователь формулирует запрос.
2. Backend формирует план изменений (preview/diff).
3. UI показывает план и запрашивает подтверждение.
4. После подтверждения backend выполняет изменения транзакционно.
5. Создается запись в истории + checkpoint для отката.
6. UI отображает результат и предоставляет “Undo”.

## 4.3 Поток отката (undo)
1. Пользователь инициирует undo.
2. Backend находит последний checkpoint/операцию.
3. Выполняет обратные изменения транзакционно.
4. Пишет событие в историю.
5. Возвращает актуальное состояние.

## 5) Ключевые зависимости и интеграции

### Внутренние
- Frontend <-> Backend по HTTP API.
- Backend <-> DB через ORM/SQL слой.
- Domain services <-> repositories (или прямой data access слой).

### Внешние
- LLM provider (например, OpenAI API) для AI-функций.
- OAuth provider (Google) для входа.
- (Опционально) Redis/queue в будущих итерациях.

Требования к интеграциям:
- timeouts, retries, circuit-breaker-подход для внешних API;
- деградация graceful fallback при частичной недоступности.

---

## 6) Доменные ограничения

1. Single-user в MVP (архитектура готова к расширению).
2. Списки по умолчанию существуют и не должны нарушать UX-поток.
3. Публичный шаринг/отправка задач — вне MVP (может быть добавлено позже).
4. Любое массовое действие AI — только через подтверждение.
5. Удаление/архивация задач должно быть обратимо в разумном окне (через history/undo).

---

## 7) Инварианты системы (нельзя нарушать)

1. **Ownership invariant**: пользователь видит и изменяет только свои сущности.
2. **Consistency invariant**: list/task/subtask связи всегда валидны.
3. **Hierarchy invariant**: иерархия подзадач ациклична.
4. **Audit invariant**: значимые изменения попадают в историю.
5. **Safety invariant**: AI не применяет изменения без explicit confirm.
6. **Rollback invariant**: после любой AI-операции доступен откат.
7. **Idempotency invariant**: повтор безопасного запроса не ломает состояние.

---

## 8) Нефункциональные требования (архитектурный уровень)

- Предсказуемая структура кода для AI-агентов.
- Явные модульные границы и контракты.
- Минимизация скрытой магии.
- Наблюдаемость: структурные логи, correlation/request id.
- Конфигурация через env и единый источник параметров.
- Миграции схемы — только контролируемо и воспроизводимо.

---

## 9) Что менять с повышенной осторожностью

- контракты API (breaking changes);
- схема БД и миграции;
- правила авторизации/доступа;
- правила выполнения AI-действий и undo-механизм;
- логика истории изменений.

Любое изменение этих зон требует отдельного описания в changelog/PR и плана отката.

---

## 10) Эволюция архитектуры (roadmap-направления)

1. Переход от single-user к multi-user без ломки доменных инвариантов.
2. Введение feature flags для рискованных фич.
3. Выделение AI-пайплайна в отдельный сервис при росте нагрузки.
4. Добавление очередей для длительных AI-задач и аналитики.
5. Усиление observability: метрики, алерты, SLO по ключевым сценариям.
