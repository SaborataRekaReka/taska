# AGENTS.md — главный README для агента

Этот документ определяет правила работы AI-агентов в репозитории. Следуй ему в каждом запуске, особенно если контекст сессии мог сброситься.

## 1) Цель документа
- Быстро восстановить рабочий контекст.
- Дать единые стандарты изменений и проверки качества.
- Снизить риск случайных/разрушительных правок.

## 2) Быстрый старт в новой сессии
При старте любой задачи агент **обязан**:
1. Прочитать этот файл полностью.
2. Проверить состояние репозитория:
   - `git status --short`
   - `git branch --show-current`
3. Понять структуру проекта:
   - `rg --files | head -n 200`
4. Найти точку входа по задаче (релевантные файлы/модули).
5. Сформировать краткий план действий и идти маленькими шагами.

## 3) Рабочий процесс
- Делай только изменения, относящиеся к запросу пользователя.
- Предпочитай минимальные и атомарные правки.
- После каждого значимого шага проверяй результат командой/сборкой/линтом.
- Не делай «рефакторинг заодно», если это не запрошено.

## 4) Что считается завершением задачи
Задача считается завершённой только если:
1. Реализованы все пункты запроса пользователя.
2. Изменения проверены релевантными командами.
3. В `git diff` нет случайных/временных файлов.
4. Изменения закоммичены с понятным сообщением.
5. Подготовлен краткий итог: что изменено и как проверено.

## 5) Обязательные проверки перед коммитом
Минимальный набор (если применимо):
- Проверка статуса изменений: `git status --short`
- Локальная проверка качества:
  - Frontend: `npm run lint` / `npm run build`
  - Backend: `npm run build`
- Если проверка не запускается из-за окружения — явно сообщи об этом в отчёте.

## 6) Политика безопасности изменений
Без явной необходимости/подтверждения **запрещено**:
- Массово удалять/переименовывать файлы.
- Менять критичные конфиги инфраструктуры (CI/CD, deployment, secrets).
- Добавлять внешние зависимости, если это не нужно для задачи.
- Вносить несовместимые API-изменения без явной фиксации в описании.

## 7) Код-стандарты
- Следуй существующему стилю проекта.
- Избегай «магии» и неочевидных абстракций.
- Пиши маленькие функции и говорящие имена.
- Не добавляй try/catch вокруг import.
- Если есть несколько вариантов решения — выбирай самый простой и поддерживаемый.

## 8) Правила работы при сбросе контекста
- Никогда не полагайся на память предыдущей сессии.
- Всегда восстанавливай контекст через файлы и git-историю.
- Перечитай текущие целевые файлы перед новой правкой.
- Повторно запусти ключевые проверки после восстановления контекста.

## 9) Коммиты
- Один логический блок изменений = один коммит.
- Формат сообщения:
  - `<type>: <краткое описание>`
  - Примеры: `docs: add AGENTS.md with workflow rules`, `fix: handle missing listId in task creation`
- Перед коммитом проверь, что не включены лишние файлы.

## 10) Шаблон финального отчёта агента
1. Что изменено.
2. Почему это сделано.
3. Какие команды запускались для проверки.
4. Результат проверок.
5. Известные ограничения/риски (если есть).

## 11) Полезные команды
- Поиск файлов: `rg --files`
- Поиск по коду: `rg "<query>" <path>`
- Проверка изменений: `git diff -- <file>`
- Проверка статуса: `git status --short`
